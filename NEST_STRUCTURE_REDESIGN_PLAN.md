# 巣の構造リライト計画（実行は別セッション）

> このファイルは、巣の「構造ジェネレーター」を作り直すための自己完結した計画書。
> 実装は別のチャット/ブランチで腰を据えて行う前提。コールドで読んでも着手できるように書いてある。
> 作成: 2026-06-17（A+B+D+E の見た目改善の続きとして合意した方向性）。

---

## 0. 一言で

巣の見た目を「実物のアリの巣の断面（土に密集したチャンバー群）」に寄せるため、**部屋・シャフトの配置を決めるジェネレーターを作り直す**。
描画レイヤー（椀状陰影・ブルードの土台・トンネル壁陰影など）と深度バンド（層）システムは**そのまま流用**する。
セーブ互換は**気にしなくてよい**（clean-slate。移行コード不要）。

参考イメージ: 実物のアリの巣断面写真（土の層・掘った椀状の部屋・ぎっしり詰まったブルード・短い通路でつながった密なチャンバー）。

---

## 1. このチャットで確定した設計判断（覆さないこと）

- **木構造をやめて、ループのあるネットワーク（メッシュ）にする。**
  - 理由: 木だと任意の2部屋の間に経路が1本しかなく、蟻の動線が固定されて見ていて単調。
  - ループ（部屋が複数の隣接部屋とつながる）にすると、蟻が複数ルートを選べて流れがばらける＝生きてる巣に見える。
- **中央の縦トランク（1本の幹）は作らない。** 動線が1本に集中して面白くないため。
- **地表の出入り口（入口の塚）は1つのまま。** 採餌ロジックが入口中心なので変えない。
  - ただし**入口のすぐ下で早めに複数ルートへ分岐**させ、内部の動線を散らす（＝「地表は穴1つ／地下はすぐ複数経路」）。
  - 将来「2つ目の入口」を強化要素にするのはアリだが、今回の必須ではない。
- **密度は一様。** 「上ほど密」はやらない（生態的には正しいがゲームでは不要との判断）。
- **深度バンド（`S.band` / 「第N層」）はそのまま活かす。** 各層＝1フロアとして、その層にクラスターを置く。層間は複数の短いトンネルでつなぐ（ループ）。
- **部屋の形は涙型/レンズ型（横長・床フラット・天井ドーム）。** アリは土の層に沿って横に掘るため。種別でサイズを変える（女王房＝大きいドーム、育児＝中、ゴミ＝小 等）。
- **トンネルは短く。** 長い放射状トンネルをやめ、部屋を密に詰めて短い枝でつなぐ。
- **数の作り込み（1層あたり何部屋／層間リンク何本など）は、作りながら調整する。** 事前に決め打ちしない。

### 新しい構造を一文で
> 入口は1つ → 地下ですぐ複数分岐 → 各深度層に密なチャンバー・クラスター → 層内・層間とも複数経路でつないでループを作る。密度は一様。

## 2. 目標構造の仕様（実装の指針）

### トポロジー
- ノード（部屋）＝グラフの頂点、トンネル＝辺。**辺はループを含む**（木ではない）。
- 入口から下へ、各深度層にチャンバーのクラスターを配置。
- 各チャンバーは**2〜3個の隣接チャンバー**とつながる（層内の横の隣＋下の層の近いチャンバー）。これでループができる。
- 「成長」＝外へ長く伸ばすのではなく、**今の層を充填しつつ、深い層を足していく**。

### 部屋（チャンバー）
- 形: レンズ/涙型。横長（幅 > 高さ）、床フラット、天井ドーム。
- 配置: クラスター内で薄い土壁を挟んで**密着**（衝突判定は「重ならないが壁は薄い」を狙う）。
- 種別とサイズ: 女王房（大・1個）、育児室（中・複数）、食料庫、休憩、兵舎、発酵室、クッキー室、貯蔵、ゴミ室。種別の割り当てルールは現行の需要ロジックを流用しつつ、層内クラスターに馴染ませる。

### トンネル
- 2クラス:
  - **降り口/層間トンネル**: 入口直下〜層をまたぐ縦寄りの接続。**複数本**（1本トランク禁止）。
  - **枝トンネル**: 層内でチャンバー同士をつなぐ短い横枝。
- 部屋の口でラッパ状に広がると自然（任意）。
- 既存の `buildEdgeStamps`（cubic bezier → スタンプ列）と E のトンネル壁陰影が**そのまま乗る**。

### 蟻の動線（面白さの源）
- グラフにループがあるので、蟻の経路は「木の親をたどる」では足りない。**最短 or 空いている経路を選ぶ**ような経路探索にする。
- これにより通り道が時間で変化＝渋滞・流れが動的＝「生きてる巣」に効く（将来の生きてる巣アニメとも整合）。

## 3. 残すもの / 書き直すもの

### 残す（再利用）
- 描画一式: `drawStaticNestSoilLayer` / `roomBowlGradient`（椀状陰影）/ ブルードの土台＆`drawBroodGrain`（`pileY`）/ トンネル壁陰影（E の2パス）/ 画面ヴィネット / 静的キャッシュ（`drawCachedStaticNestLayer`・`getNestVisualSignature`）。
- ノード/辺のデータ表現とその描画: blob（`buildNodeBlobAndSlots` / `blobParamsForType` / `fillBlobPath`）、edge の bezier＋`buildEdgeStamps`＋`drawStamps`。
- 深度バンド（`S.band`：index/top/bottom、「第N層」降下）。
- 入口・採餌まわり（地表の出入り口は1つのまま）。
- 部屋の在庫/種別/容量ロジック（`recalc` の caps、種別ごとの需要）。

### 書き直す
- **配置ジェネレーター**（本丸・load-bearing）:
  - `getDigTarget()`（部屋/シャフトを作る唯一の場所。`forcedType`、層内配置、衝突回避）。
  - `expandMap()` / `forceExpandRoom()` / メインシャフトのキュー処理。
  - 建築アリの割り当て `rebuildBuilderAssignments()` / `buildAssignments`（新トポロジーに合わせる）。
  - 部屋・トンネルの**初期生成**（新規ゲーム時の巣の作り方）。
- **蟻の経路探索**: 木前提（親辺をたどる類）ならループ対応へ。最短/混雑回避のルート選択。
  - 現行の移動タスク（`go_q` / `go_n` / `go_larva_pick` 等）と `move()` のエッジ走行を確認し、ループでも破綻しないように。
- **セーブ/ロード形式**: 新しいノード/辺/接続を保存できるように（旧形式は捨ててよい）。`G.toSave` / `S.serializeWorld` 周辺。

## 4. 実装アプローチ

1. **devトグルで新ジェネレーターを並走**させる。既存生成と切り替えて見比べられるようにし、本番セーブには影響させない。
   - 例: `S._devMode` 下のボタン or URL パラメータで「新生成」を選ぶ。新規ワールドを新ジェネで作って描画確認。
2. 新トポロジーが見た目・動線とも良ければ、**正式採用してデータ構造を新形式へ**。旧生成と旧セーブは破棄。
3. 仕上げに描画パラメータ（クラスター間隔・壁の薄さ・トンネル幅）を実機で微調整。

### フェーズ分け（目安）
- **F1 生成の骨格**: 入口→複数降り口→各層にクラスター（仮の矩形/円充填）。ループ辺を張る。描画は既存流用でとりあえず出す。
- **F2 形の作り込み**: レンズ/涙型チャンバー、薄壁の密着、トンネル短縮・ラッパ口。
- **F3 蟻の経路**: ループ対応の経路選択。動線がばらけるか実機確認。
- **F4 建築AI/成長**: 既存層の充填＋深い層追加のルールを新トポロジーに合わせる。
- **F5 セーブ新形式 & devトグル撤去**: 正式採用。

## 5. 作りながら詰めるパラメータ（決め打ちしない）

- 1層あたりのチャンバー数（密度感）。
- 1チャンバーが持つ隣接リンク数（ループの気持ちよさ）。
- 降り口（入口直下の分岐）の本数。
- 層間トンネルの本数。
- クラスター間の壁の薄さ（衝突距離）。
- トンネル幅・部屋サイズの種別差。

## 6. リスク・検証

- **リスク**: 配置ジェネレーターは衝突回避・dig-edge・建築割り当てが絡む load-bearing。蟻の経路探索のループ対応は破綻すると蟻が固まる/暴走する恐れ。→ devトグル並走で旧実装を壊さずに進める。
- **検証**: ヘッドレスプレビューは rAF が止まる/hidden 固着でスクショ不可なことがある。**実ブラウザ（`localhost:8765`）での目視確認**を基本にする。内部関数は IIFE スコープ（`G`/`S` のみグローバル）なので、効果は実UI操作＋`G`/`S` 読みで確認。
- 性能: ループでエッジ数が増えると静的キャッシュの再構築コストが上がりうる。大きい巣で要確認（`getNestVisualSignature` のハッシュ対象に新接続が乗るか含め）。
